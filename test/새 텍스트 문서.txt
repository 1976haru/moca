<project>
  <name>통합콘텐츠 생성기</name>
  <goal>
    기존 대본생성기와 기존 음성·영상 프롬프트 생성기를 기반으로,
    자동숏츠 프로그램과 콘텐츠 생성기를 추가하여
    하나의 통합형 콘텐츠 제작 웹앱으로 재구성한다.
  </goal>

  <core_principle>
    새로 백지에서 만드는 것이 아니라,
    기존 기능을 누락 없이 유지하면서
    무거운 단일 HTML 구조를 모듈형 구조로 분리하고
    통합 허브를 추가하는 방향으로 설계 및 구현한다.
  </core_principle>

  <user_profile>
    사용자는 기획은 가능하지만 코딩은 거의 모른다.
    따라서 설명은 쉽고 명확해야 하며,
    복잡한 기술 용어를 남발하지 말고
    "무엇을 왜 하는지"를 단계별로 설명해야 한다.
  </user_profile>

  <must_keep>
    <item>기존 대본생성기의 핵심 기능을 누락 없이 유지</item>
    <item>기존 음성·영상 프롬프트 생성기의 핵심 기능을 누락 없이 유지</item>
    <item>자동숏츠 연결 기능 추가</item>
    <item>콘텐츠 생성기 기능 추가</item>
    <item>한국어 우선 개발, 이후 일본어/영어 확장 가능 구조</item>
    <item>스마트폰, 태블릿, PC 반응형 지원</item>
    <item>향후 판매 가능한 수준의 구조와 가이드 고려</item>
  </must_keep>

  <product_scope>
    <common_core>
      <item>프로젝트 생성/저장</item>
      <item>콘텐츠 유형 선택: 숏츠, 블로그, 뉴스레터, 웹툰, 대화형, 감동사연, 코믹, 티키타카 등</item>
      <item>언어 선택: 한국어, 일본어, 영어</item>
      <item>자료 입력: 키워드, 문서, URL, 뉴스 링크, 유튜브 링크, 이미지, 영상</item>
      <item>결과 저장: 대본, 프롬프트, 음성 설정, 이미지 설정, 영상 설정, 내보내기</item>
    </common_core>

    <script_engine>
      <item>기존 대본생성기 기능 100% 유지</item>
      <item>테마별 모듈 분리: 시니어, 코믹, 감동사연, 뉴스형, 대화형 등</item>
      <item>모델 선택 가능: OpenAI, Claude, Gemini</item>
      <item>구조화 출력(JSON) 우선</item>
      <item>스타일, 길이, 후킹, CTA 규칙 세트 지원</item>
    </script_engine>

    <media_engine>
      <item>TTS: ElevenLabs 우선</item>
      <item>이미지: OpenAI, Gemini, 기타 공급자 교체 가능 구조</item>
      <item>음악: Suno 또는 대체 공급자</item>
      <item>영상: InVideo 연동</item>
      <item>완전자동이 어려운 경우 반자동용 상세 프롬프트 및 장면별 가이드 자동 생성</item>
    </media_engine>

    <auto_shorts_engine>
      <item>장면 분할</item>
      <item>언어별 폰트 템플릿</item>
      <item>채널/카테고리별 보이스 자동 선택</item>
      <item>썸네일, 제목, 설명, 태그 생성</item>
      <item>업로드용 메타데이터 생성</item>
      <item>가능하면 유튜브 업로드 연동 고려</item>
    </auto_shorts_engine>

    <content_engine>
      <item>블로그, 뉴스레터, 웹툰, 링크형 콘텐츠 대응</item>
      <item>이미지, 영상, 뉴스 링크, 유튜브 링크와 연계</item>
      <item>실시간 트렌드 반영 기능 확장 가능 구조</item>
    </content_engine>
  </product_scope>

  <design_requirements>
    <item>Claude Design을 활용하기 쉬운 구조로 설계</item>
    <item>UI는 직관적이고 초보자 친화적이어야 함</item>
    <item>모바일/태블릿/PC 반응형 필수</item>
    <item>한국/일본 폰트 선택 구조 고려</item>
    <item>카테고리별 UX 분기 가능 구조</item>
  </design_requirements>

  <architecture_rules>
    <item>거대한 단일 HTML 파일 구조를 금지</item>
    <item>기능별 모듈 구조로 분리</item>
    <item>카테고리 로직은 독립 모듈화</item>
    <item>공통 상태 관리와 공통 유틸을 분리</item>
    <item>API 공급자별 어댑터 구조를 사용</item>
    <item>공통 코어와 개별 엔진을 분리</item>
    <item>향후 기능 추가 시 특정 모듈만 수정 가능해야 함</item>
  </architecture_rules>

  <provider_strategy>
    <item>특정 AI 서비스 하나에 종속되지 않게 설계</item>
    <item>스크립트 모델, 이미지 모델, 음성 모델을 각각 교체 가능해야 함</item>
    <item>환경변수와 설정 파일로 공급자 변경 가능해야 함</item>
  </provider_strategy>

  <automation_policy>
    <item>100% 자동화가 가능한 영역은 자동화</item>
    <item>100% 자동화가 어려운 영역은 반자동 UX를 강화</item>
    <item>반자동일 경우 사용자가 바로 복사·붙여넣기 가능한 상세 프롬프트를 생성</item>
    <item>작업 실패 시 대체 경로를 제시</item>
  </automation_policy>

  <token_efficiency_rules>
    <item>한 번에 전체 구현하지 말고 단계별로 진행</item>
    <item>항상 먼저 설계안을 요약한 뒤 구현</item>
    <item>출력은 불필요한 장문 설명보다 구조화된 형식 우선</item>
    <item>이미 합의된 내용은 반복 설명하지 말 것</item>
    <item>한 번의 응답에서는 현재 단계에 필요한 코드와 설명만 제공</item>
    <item>전체 파일이 꼭 필요하지 않다면 수정 대상 파일과 변경 이유를 먼저 요약</item>
  </token_efficiency_rules>

  <security_and_sales>
    <item>향후 판매를 고려한 구조로 설계</item>
    <item>API 키는 클라이언트에 직접 노출하지 않는 구조를 우선 고려</item>
    <item>사용자 권한, 프로젝트 저장, 접근 제어를 확장 가능하게 설계</item>
    <item>완전한 복제 방지는 어렵지만 무단 복사 난이도를 높이는 구조를 고려</item>
    <item>사용자용 가이드와 운영자용 가이드를 분리 가능하게 설계</item>
  </security_and_sales>

  <non_negotiable_rules>
    <item>기존 기능을 임의로 삭제하지 말 것</item>
    <item>모호하면 기능을 빼지 말고 유지한 상태로 리팩터링할 것</item>
    <item>디자인보다 구조 안정성과 유지보수성을 우선할 것</item>
    <item>초보자가 관리 가능한 방식으로 설명할 것</item>
  </non_negotiable_rules>

  <working_method>
    <step>먼저 현재 요구사항을 기준으로 전체 구조를 재정리한다</step>
    <step>그 다음 권장 폴더 구조를 제안한다</step>
    <step>그 다음 기능 분해표를 만든다</step>
    <step>그 다음 화면 구조를 설계한다</step>
    <step>그 다음 구현 순서를 제안한다</step>
    <step>승인 없이 한 번에 거대한 코드 덤프를 출력하지 말고 단계별로 진행한다</step>
  </working_method>

  <required_output_format>
    <section>1. 이해한 목표</section>
    <section>2. 전체 아키텍처 제안</section>
    <section>3. 폴더 구조 제안</section>
    <section>4. 기능 모듈 분해</section>
    <section>5. UI 화면 구성</section>
    <section>6. 1차 구현 범위</section>
    <section>7. 이후 구현 순서</section>
    <section>8. 지금 바로 생성할 산출물</section>
  </required_output_format>

    구조로 재설계해야 한다.
  </integration_direction>

</project>
